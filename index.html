Redis集群

当使用网络分区的时候就会涉及数据的一致性，根据CAP定理：当网络分区发生时，一致性和可用性两难全，如何实现主节点与从节点的一致性就成为了难题，Redis通过保持最终一致的方式，实现主从数据异步同步，节点会努力追赶主节点，最终从节点的状态和主节点的状态保持一致。

增量同步

redis同步的是指令流，主节点会将哪些对自己的状态产生修改性影响的指令记录在本地的Buffer中，然后异步将buffer中的指令同步到从节点，从节点执行同步指令，主节点反馈自己同步到哪里
缺点：buffer是有限的，如果buffer内容太多，之后的指令则会覆盖前面的指令

快照同步

快照同步，主节点执行一次bgsave，将当前的内存数据全部存放到磁盘文件中，然后再将快照文件的内容全部传送回从节点

Sentinel

Sentinel负责监控redis集群的健康情况，如果一个节点挂了，集群可以正常运作

如果其中一个节点挂了，则会重新连接IP，查询数据的时候使用轮询的方式。
具体可以参考阿里Sentinel

Codis

在大数据高并发的场景下，单个的redis实例往往会比较受限制，限制在于redis的内存使用，而且单个redis的内存不宜过大，如果太大容易导致rdb文件过大，进一步导致主从同步的时候全量同步时间过长，在实例重启恢复时也会消耗很长的时间。所以应运而生出很多集群方案，类似codis

codies而且是无状态的，所以可以启动多个codies来支撑高QPS的场景
codies默认将所有的key划分为1024个槽位，首先对客户端的key进行crc32位运算得到hash值，然后对hash得到的整数对1024取余，这个余数就是对应的key的槽位，之后通过zookeeper方式维护一致性

Cluster

Cluster集群使用了三个redis节点组成，他们之间通过一种特殊的二进制协议交互集群信息，与codies不同的是cluster使用了16384个槽位

Redis的基本使用

Redis中的位图

在日常的开发过程中，经常会涉及一部分的bool类型的存储，如果使用关系性数据库，则会产生很大的数据量，比如需要统计一个公司所有人365的打卡情况，则需要 n*365，这时候需要的数据量是巨大的，而redis中产生了位图的概念，类似byte数组。

HyperLogLog

如果需要统计页面的UV和PV数据，怎么处理？（UV表示独立用户unique，PV表示page view）
使用Redis的set集合，对独立的IP进行去重，则可以获取UV
可是如果用户量特别大，去重又特别消耗资源，这时候由于set引申出了HyperLogLog

使用Python脚本测试HyperLogLog误差

布隆过滤器

当需要时候进行大量数据去重的时候，如果使用关系型数据库，就会频繁的使用exist查询，这样对性能消耗很大，而Bloom Filter的时候可以节约大量的空间
Bloom Filter的原理是将字符串进行Hash处理，然后将其值映射到数组上去
PS：需要Redis 4.0以上的版本处理，日工bfadd bfexists命令

限流

当应用的访问数据量特别大的时候，一般需要限制用户的行为，这时候就需要使用zset这个数据结构

zset维护了一个score的数据结构，我们可以使用时间维护一个窗口，score表示在哪个时间发生的行为，如果不是在这个窗口内的，则剔除

漏斗限流算法

通过计算剩余空间大小来判断是否可以继续

Redis

redis 数据结构

redis的所有数据结构都是通过key/value的形式组成的，所有的数据结构都以唯一的key字符串作为名称，然后通过这个唯一的key值来获取相应的value。不同的数据结构差异在于value的结构不一样。

String类型

Redis的字符串可以用于缓存用户信息，可以将用户信息结构体用Json序列化成字符串，然后将序列化后的字符串塞进Redis进行缓存，而取信息的时候经过一次反序列化的过程。
Redis的字符串采用了预分配的方式，这样可以减少内存的频繁分配，类似Java中的ArrayList，也就是为当前字符分配的空间一般高于实际字符串长度len

List类型

Redis的列表相当于Java语言中的LinkedList，是链表的形式，支持双指针的前后遍历，由于是链表，其插入和删除的速度特别快（具体链表为什么插入和删除特别快是由于链表不需要读取内部的值）
Redis列表结构经常作为异步队列使用

Hash类型

在HashMap在字典很大的时候，Rehash会十分耗时，Redis中为了追求高性能，通常采用渐进式rehash策略，渐进式策略会在rehash的同时，保留两个hash结构，在查询的时候会对两个都进行查看，在后续的定时任务中慢慢的将hash的内容一步步迁移过去，当迁移完成的时候，就会使用新的hash结构代替

Set类型

set类型相当于Java中的HashSet，它内部的键值对是无序的、唯一的，所以可以用来存储某活动的中奖ID，保证唯一性

Zset列表

zset类型一方面是一个set，可以保证内部的value的唯一性，另一方面它可以给每一个value赋予一个score，代表这个value的排序权重， zset内部是以跳跃表的形式

Redis分布式锁

分布式应用通常涉及到并发问题，由此产生了分布式锁的方式
分布式锁：setnx(set if not exist)
